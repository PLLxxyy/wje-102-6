# 菜谱共创工坊 — 项目提示词

> 项目代号：wje-102 | COMPOSE_PROJECT_NAME：wjerecipeworks

---

## 一、硬性约束

1. **前后端分离**：前端 React + TypeScript + Ant Design + Vite，后端 Node.js + NestJS + TypeORM，数据库 MySQL 8.0。三者各自独立容器，通过 Docker Compose 编排。
2. **端口固定**：前端 `28602:80`，后端 `29602:3000`，数据库 `57602:3306`。任何环境不得擅自修改端口映射。
3. **禁止硬编码连接信息**：数据库连接串、JWT 密钥等敏感信息一律通过 `.env` 文件注入，代码中不得出现明文密码或密钥。
4. **TypeScript strict 模式**：前后端均开启 `strict: true`，禁止使用 `any` 类型（经审批的极少数场景除外，需写注释说明原因）。
5. **RESTful API 规范**：所有接口遵循 RESTful 风格，统一返回格式 `{ code: number, data: any, message: string }`。
6. **ORM 禁止裸 SQL**：所有数据库操作通过 TypeORM Repository / QueryBuilder，禁止拼接原始 SQL。
7. **组件文件单一职责**：每个 `.ts` / `.tsx` 文件只承担一种职责（组件 / service / controller / entity / hook），严禁合并。
8. **Git 规范**：提交信息使用 Conventional Commits 格式（`feat:`, `fix:`, `chore:` 等）。

## 二、项目主要需求

### 2.1 项目概述

菜谱共创工坊是一个面向美食爱好者的全栈 Web 应用。用户可以创建、编辑、分享菜谱，支持多人实时协作编辑（类似 Google Docs 体验）。系统内置食材营养数据库，可根据食材用量自动计算菜谱的热量和营养素。用户可以管理个人收藏夹，按分类、难度等维度发现优质菜谱。

### 2.4 横切关注点

#### 横切关注点 1：认证授权（JWT）

**触达层**：DB users 表 → 后端 auth 模块（JwtStrategy, JwtAuthGuard）→ 后端各 Controller 装饰器 → 前端 axios 拦截器（自动附加 token、处理 401）→ 前端 PrivateRoute 路由守卫 → 前端 useAuth hook。

- 注册接口：`POST /api/auth/register`，返回 JWT token
- 登录接口：`POST /api/auth/login`，返回 JWT token
- 获取当前用户：`GET /api/auth/profile`，需要 JWT
- 前端 token 存储于 localStorage，axios 拦截器自动附加 `Authorization: Bearer <token>`
- 前端 401 响应自动清除 token 并跳转登录页
- 路由分为公开路由和受保护路由，受保护路由需登录后访问

#### 横切关注点 2：操作日志

**触达层**：DB operation_logs 表 → 后端 LoggingInterceptor（拦截所有请求，记录操作人、动作、资源）→ 后端 OperationLogService（持久化）→ 前端日志管理页面（仅 admin 可见）。

- 记录所有写操作（POST/PUT/DELETE）的操作人、动作、资源、IP
- 读操作不记录
- admin 用户可在前端查看操作日志列表，支持按时间、用户、动作筛选

### 2.9 技术要求

- **前端状态管理**：使用 Zustand 管理全局状态，按实体拆分 store
- **前端路由**：React Router v6，受保护路由使用 PrivateRoute 包裹
- **前端 HTTP**：axios 封装统一请求工具（`frontend/src/utils/request.ts`），配置 baseURL、超时、拦截器
- **后端框架**：NestJS 模块化架构，每个业务实体一个 Module
- **后端 ORM**：TypeORM，entity 使用装饰器定义，开启 synchronize 自动建表（开发环境）
- **后端验证**：使用 class-validator + class-transformer 做 DTO 校验
- **WebSocket**：NestJS 内置 @WebSocketGateway，用于协作编辑的实时同步
- **密码加密**：bcrypt，salt rounds = 10
- **JWT 签发**：过期时间 7 天，密钥从环境变量读取
- **文件上传**：封面图、步骤图片暂用 URL 方式（后续可扩展本地/OSS 上传）

### 严禁合并职责到单一文件

- **每个 entity 单独一个文件**，不得在一个文件中定义多个 entity
- **每个 service 单独一个文件**，不得合并多个业务的 service
- **每个 controller 单独一个文件**，不得合并多个业务的 controller
- **每个 hook 单独一个文件**，不得在单个文件中混写多个 hook
- **每个页面组件单独一个文件**，不得在一个文件中导出多个页面
- **每个 store 单独一个文件**，不得在一个文件中定义多个 store
- **每个 API 模块单独一个文件**，按实体拆分

## 四、核心流程

### 4.1 用户注册 / 登录流程

1. 用户填写注册表单 → 前端调用 `POST /api/auth/register`
2. 后端校验用户名/邮箱唯一性 → bcrypt 加密密码 → 写入 users 表 → 签发 JWT 返回
3. 前端存储 token 到 localStorage → useAuth hook 更新状态 → 跳转首页
4. 登录流程：`POST /api/auth/login` → 校验密码 → 签发 JWT → 后续请求 axios 拦截器自动附加 Authorization header

### 4.2 创建菜谱流程

1. 用户进入 `/recipe/create` 页面
2. 填写菜谱基本信息（标题、简介、分类、几人份、难度、封面图）
3. 使用 IngredientSelector 组件从食材库搜索并选择食材，填写用量和单位
4. 使用 StepEditor 组件添加烹饪步骤（描述、图片、预计耗时），支持拖拽排序
5. 提交时前端调用 `POST /api/recipes`，后端同时创建 Recipe、RecipeStep、RecipeIngredient 记录
6. 营养自动计算：后端根据食材用量 × 食材营养数据（/100g），计算总热量/蛋白质/脂肪/碳水，写入 Recipe 的营养字段
7. 初始状态为 draft，用户可手动切换为 published

### 4.3 多人协作编辑流程

1. 菜谱作者在详情页点击「邀请协作者」→ 搜索用户 → 发送邀请（创建 Collaboration 记录，role=editor）
2. 被邀请用户接受邀请 → Collaboration.accepted_at 赋值
3. 协作者进入 `/recipe/:id/edit` → useWebSocket hook 建立 WebSocket 连接
4. 编辑操作（修改标题、添加食材、编辑步骤等）通过 WebSocket 实时推送给所有在线协作者
5. 后端 CollaborationGateway 管理房间（以 recipe_id 为房间号），广播变更事件
6. 保存时创建新版本快照

### 4.4 营养自动计算流程

1. 创建/编辑菜谱时，前端 useNutritionCalc hook 接收 `[{ ingredientId, amount, unit }]` 数组
2. 从 IngredientStore 中获取每个食材的营养数据（calories_per_100g 等）
3. 计算公式：`营养素总量 = (amount / 100) × per_100g`（假设 amount 单位为 g）
4. 前端实时预览 → 提交时后端再次计算并持久化到 Recipe 记录

### 4.5 版本历史流程

1. 每次保存菜谱时，后端将当前菜谱快照（含步骤和食材）序列化为 JSON，存入 recipe_versions 表
2. 用户在菜谱详情页点击「版本历史」→ 前端调用 `GET /api/recipes/:id/versions`
3. 版本列表按时间倒序展示
4. 点击任意版本可查看该版本详情，支持两个版本对比差异（字段级 diff）
5. 点击「回滚到此版本」→ 后端将该版本数据覆盖当前菜谱 → 创建新版本快照

### 4.6 收藏夹管理流程

1. 用户在菜谱详情页点击「收藏」→ 选择或创建收藏夹 → 前端调用 `POST /api/collections/:id/recipes`
2. 在 `/collections` 页面管理收藏夹：创建/编辑/删除收藏夹，移除已收藏菜谱
3. 支持将收藏夹设为公开/私有

## 六、代码存放规则

1. **根目录结构**：项目根目录下仅有 `frontend/`、`backend/`、`docker-compose.yml`、`.env.example`、`.env`、`.gitignore`，不得在根目录放置业务代码文件。
2. **前端代码**：所有前端代码存放在 `frontend/` 目录下，遵循第三节的文件结构清单。
3. **后端代码**：所有后端代码存放在 `backend/` 目录下，遵循第三节的文件结构清单。
4. **前端公共组件**：所有被 2 个及以上页面使用的组件，必须放在 `frontend/src/components/common/`。
5. **前端 hooks**：所有自定义 hooks 放在 `frontend/src/hooks/`，命名以 `use` 开头。
6. **前端 API 层**：所有 HTTP 请求封装在 `frontend/src/api/`，页面组件中禁止直接调用 axios。
7. **前端类型定义**：所有共享类型放 `frontend/src/types/`，entity 对应的类型按文件拆分。
8. **后端模块**：每个业务实体一个 Module 文件夹，包含 controller / service / entity / dto / module 文件。
9. **后端公共**：guards / decorators / pipes / filters 统一放在 `backend/src/common/`。
10. **配置文件**：`docker-compose.yml`、`.env.example` 放根目录；`Dockerfile` 分别放在 `frontend/` 和 `backend/`；`nginx.conf` 放 `frontend/`。
11. **种子数据**：初始化数据脚本放在 `backend/src/utils/seed.ts`，通过 `npm run seed` 命令执行。
